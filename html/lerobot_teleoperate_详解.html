1. 整体架构概览

遥操作系统的核心思想是：操作者通过一个 Leader 设备（遥操作器）发出动作指令，Follower 机器人实时跟随执行。整个系统采用三层架构：

┌──────────────────────────────────────────────────────┐
│                    CLI Layer                          │
│  draccus + parser.wrap() → TeleoperateConfig         │
├──────────────────────────────────────────────────────┤
│                   Domain Layer                        │
│  ┌──────────────┐    ┌──────────────┐                │
│  │ Teleoperator │    │    Robot     │                │
│  │  (Leader)    │    │  (Follower)  │                │
│  │  get_action()│    │ send_action()│                │
│  └──────┬───────┘    └──────┬───────┘                │
│         │                   │                        │
│  ┌──────┴───────────────────┴───────┐                │
│  │       Processor Pipelines        │                │
│  │  (teleop → robot action bridge)  │                │
│  └──────────────────────────────────┘                │
├──────────────────────────────────────────────────────┤
│                   Loop Layer                          │
│  teleop_loop(): 读取 → 处理 → 发送 → 显示（循环）     │
└──────────────────────────────────────────────────────┘

文件位置： lerobotteleoperate.py

2. 入口与配置解析

2.1 main() 函数（第 248-251 行）

def main():
    register_third_party_plugins()  # 注册第三方插件
    teleoperate()                    # 启动遥操作

这是程序的真正入口，做了两件事：

  1. 注册第三方插件：调用 registerthirdparty_plugins() 加载外部插件包，允许第三方扩展新的机器人类型和遥操作器类型（通过 Python 的包发现机制）。
  2. 调用遥操作主函数：进入 teleoperate()。

2.2 @parser.wrap() 装饰器（第 208 行）

@parser.wrap()
def teleoperate(cfg: TeleoperateConfig):

parser.wrap() 是 LeRobot 对 draccus 库的封装（见 configs/parser.py），它做了三件关键的事：

  1. 自动解析 CLI 参数：将命令行参数（如 --robot.type=so101_follower）自动映射到 TeleoperateConfig 数据类的嵌套字段上。
  2. 加载插件：在解析配置之前，先扫描 CLI 中的 --*.discoverpackagespath= 参数，加载指定的第三方插件包，使得插件的配置子类能被 draccus 识别。
  3. 支持 Hub 加载：如果提供了 --configpath 参数且配置类有 frompretrained 方法，则从 Hugging Face Hub 加载预训练配置。

draccus 的核心能力是：通过 --robot.type=so101_follower 这样的 CLI 参数，自动从注册的子类中选择正确的 RobotConfig 子类（如 SOFollowerRobotConfig）来实例化。

3. 核心配置：TeleoperateConfig

位置： 第 109-124 行

@dataclass
class TeleoperateConfig:

字段详解

字段 类型 默认值 说明
teleop TeleoperatorConfig 必填 Leader 设备的完整配置（类型、端口、ID、相机等）
robot RobotConfig 必填 Follower 机器人的完整配置（类型、端口、ID、相机等）
fps int 60 控制循环的目标频率，循环会尝试以这个频率运行

TeleoperatorConfig 和 RobotConfig 都是通过 draccus 的多态机制实现的可扩展配置基类，具体类型由 CLI 的 .type 参数决定。

4. 主控制循环：teleop_loop

位置： 第 127-205 行

这是整个遥操作系统的核心，是一个固定频率的实时控制循环。

4.1 函数签名

def teleop_loop(

    robot: Robot,                                    # Follower 机器人实例

    teleop_action_processor: RobotProcessorPipeline,  # 遥操作动作处理器
    robot_action_processor: RobotProcessorPipeline,   # 机器人动作处理器
    robot_observation_processor: RobotProcessorPipeline, # 观测处理器

│           │                                  │

4.3 关键设计细节

(a) 观测获取（第 164 行）

obs = robot.get_observation()

在每轮循环开始时先获取机器人的当前观测（包括关节角度、速度、相机图像等）。注释明确指出：目前主要用于可视化，默认的遥操作动作处理器（IdentityProcessorStep）可以接受 None 作为观测输入。

(b) Unitree G1 的特殊处理（第 166-167 行）

if robot.name == "unitree_g1":
    teleop.send_feedback(obs)

对于 Unitree G1 这种人形机器人，遥操作器需要接收来自机器人的力反馈数据，实现双向控制。这是唯一有此需求的机器人类型。

(c) 三级处理器管道（第 170-176 行）

raw_action = teleop.get_action()                                     # 从 Leader 读取原始动作
teleop_action = teleop_action_processor((raw_action, obs))           # 处理遥操作动作
robot_action_to_send = robot_action_processor((teleop_action, obs))  # 转换为机器人动作

这是整个架构中最精妙的设计——三级处理管道：

管道 输入 输出 默认行为
teleopactionprocessor (raw_action, obs) 处理后的动作 恒等变换
robotactionprocessor (teleop_action, obs) 机器人可执行动作 恒等变换
robotobservationprocessor obs 处理后的观测 恒等变换

默认的三个处理器都是 IdentityProcessorStep（恒等变换），意味着 Leader 的动作直接映射到 Follower。但系统预留了扩展点，允许：

  • 动作空间映射：Leader 和 Follower 关节结构不同时进行映射
  • 归一化/反归一化：不同数值范围之间的转换
  • Delta 动作转换：从绝对位置转为增量位置

(d) 可视化（第 181-196 行）

if display_data:
    obs_transition = robot_observation_processor(obs)
    log_rerun_data(observation=obs_transition, action=teleop_action, ...)
    # 终端打印动作值
    for motor, value in robot_action_to_send.items():
        print(f"{motor:<{display_len}} | {value:>7.2f}")
    move_cursor_up(len(robot_action_to_send) + 3)
  • 使用 Rerun（一个实时可视化工具）显示观测和动作数据
  • 终端中使用 movecursorup 实现原地刷新效果，避免滚动刷屏
  • 支持本地和远程 Rerun 服务器

(e) 精确帧率控制（第 199 行）

dt_s = time.perf_counter() - loop_start
precise_sleep(max(1 / fps - dt_s, 0.0))

使用 precisesleep 而非 time.sleep：

  • spin_threshold=10ms：剩余时间 ≤10ms 时，使用忙等待（自旋）保证精度
  • sleep_margin=5ms：需要 sleep 超过 5ms 时，提前 5ms 唤醒再用自旋补足
  • 优先保证时序精度而非 CPU 占用率

(f) 超时控制（第 204-205 行）

if duration is not None and time.perf_counter() - start >= duration:
    return

5. 初始化和编排：teleoperate

位置： 第 208-245 行

@parser.wrap()
def teleoperate(cfg: TeleoperateConfig):
    # 1. 初始化日志
    init_logging()
    
    # 2. 初始化 Rerun 可视化（如果启用）
    if cfg.display_data:
        init_rerun(session_name="teleoperation", ip=cfg.display_ip, port=cfg.display_port)
    
    # 3. 远程显示时自动启用图像压缩
    display_compressed_images = (
        True if (cfg.display_data and cfg.display_ip is not None and cfg.display_port is not None)
        else cfg.display_compressed_images
    )
    
    # 4. 工厂方法创建实例
    teleop = make_teleoperator_from_config(cfg.teleop)  # 创建 Leader
    robot = make_robot_from_config(cfg.robot)            # 创建 Follower
    
    # 5. 创建默认处理器管道
    teleop_action_processor, robot_action_processor, robot_observation_processor = make_default_processors()
    
    # 6. 连接设备
    teleop.connect()
    robot.connect()
    
    # 7. 运行主循环（带异常保护）
    try:
        teleop_loop(...)
    except KeyboardInterrupt:
        pass  # 优雅地处理 Ctrl+C
    finally:
        # 8. 清理资源
        if cfg.display_data:
            rr.rerun_shutdown()
        teleop.disconnect()
        robot.disconnect()

关键步骤解析

工厂方法模式

teleop = make_teleoperator_from_config(cfg.teleop)
robot = make_robot_from_config(cfg.robot)

这两个工厂函数根据配置中的 .type 字段动态选择正确的子类：

  • --teleop.type=so101_leader → 创建 SOLeader 实例
  • --robot.type=so101_follower → 创建 SOFollower 实例

这种设计使得系统高度可扩展——添加新机器人只需实现 Robot 子类并注册即可。

资源管理

使用 try/finally 确保无论正常退出还是异常中断，都会：

  1. 关闭 Rerun 可视化连接
  2. 断开 Teleoperator 连接（释放串口等资源）
  3. 断开 Robot 连接（释放串口等资源）

6. 抽象基类：Robot 与 Teleoperator

6.1 Robot 抽象基类

位置： robots/robot.py（212 行）

定义了所有 LeRobot 兼容机器人的标准接口：

抽象方法 说明
observation_features 返回观测数据的结构描述（类型和形状）
action_features 返回动作数据的结构描述
is_connected 是否已连接
connect(calibrate=True) 建立通信连接，可选自动校准
is_calibrated 是否已校准

send_action(action) 发送动作指令 → RobotAction
disconnect() 断开连接并清理资源

校准机制：

  • 校准数据以 JSON 格式存储在 ~/.cache/huggingface/lerobot/calibration/robots/{name}/{id}.json
  • 包含每个电机的偏移量、方向等 MotorCalibration 数据

6.2 Teleoperator 抽象基类

位置： teleoperators/teleoperator.py（209 行）

与 Robot 接口高度对称，但语义略有不同：

抽象方法 说明
action_features 返回产生的动作的结构描述
feedback_features 返回期望的反馈数据的结构描述
is_connected 是否已连接
connect(calibrate=True) 建立通信连接
is_calibrated 是否已校准
calibrate() 执行校准
configure() 运行时配置
get_action() 读取操作者的动作 → RobotAction

disconnect() 断开连接

6.3 Robot 与 Teleoperator 的异同

                Robot (Follower)              Teleoperator (Leader)
                ─────────────────             ────────────────────
数据流向:       接收动作 → 执行                读取操作者 → 产生动作
核心方法:       send_action(action)           get_action()
辅助方法:       get_observation()             send_feedback(obs) [可选]
校准:           电机零点/方向校准              电机零点/方向校准
共性:           都通过串口/USB 与 Feetech 等电机总线通信

7. 处理器管道系统

位置： processor/ 目录

7.1 设计动机

Leader 和 Follower 可能具有：

  • 不同的关节数量和名称（如 5 关节 Leader → 6 关节 Follower）
  • 不同的数值范围（如 Leader 返回 [-1,1]，Follower 需要 [0,4095]）
  • 不同的动作模式（绝对位置 vs 增量位置）

处理器管道负责在这些差异之间架设桥梁。

7.3 默认处理器

位置： processor/factory.py

def make_default_processors():
    teleop_action_processor = make_default_teleop_action_processor()
    robot_action_processor = make_default_robot_action_processor()
    robot_observation_processor = make_default_robot_observation_processor()
    return (teleop_action_processor, robot_action_processor, robot_observation_processor)

三个默认处理器都只包含一个 IdentityProcessorStep（不做任何变换），意味着在最简单的情况下（如同构的 SO Leader/Follower 对），动作从 Leader 直接传递到 Follower。

7.4 可用的 ProcessorStep 类型

处理器 功能
IdentityProcessorStep 恒等变换（什么都不做）
NormalizerProcessorStep 数据归一化
UnnormalizerProcessorStep 数据反归一化
MapDeltaActionToRobotActionStep Delta 动作 → 绝对位置
MapTensorToDeltaActionDictStep 张量格式 → Delta 字典
VanillaObservationProcessorStep 标准观测处理
DeviceProcessorStep 设备间数据传输（CPU↔GPU）
AddBatchDimensionProcessorStep 添加批次维度
RenameObservationsProcessorStep 重命名观测字段
ActionTokenizerProcessorStep 动作分词
ImageCropResizeProcessorStep 图像裁剪/缩放

所有 ProcessorStep 通过 ProcessorStepRegistry 注册，支持按名称动态实例化。

8. 具体实现案例：SO-100/101

8.1 SOFollower（Follower 机器人）

位置： robots/sofollower/sofollower.py

关键特征:
- 继承自 Robot 基类
- 使用 FeetechMotorsBus 通过串口与 STS3215 舵机通信
- 6 个关节: shoulder_pan, shoulder_lift, elbow_flex, wrist_flex, wrist_roll, gripper
- 支持度数和归一化范围两种模式
- 可集成相机（OpenCV、RealSense、ZMQ）

8.2 SOLeader（Leader 遥操作器）

位置： teleoperators/soleader/soleader.py

关键特征:
- 继承自 Teleoperator 基类
- 同样使用 FeetechMotorsBus + STS3215 舵机
- 5 个关节: shoulder_pan, shoulder_lift, elbow_flex, wrist_flex, wrist_roll
- 读取关节角度作为动作输出（操作者手动拖拽 Leader 臂）

8.3 Leader-Follower 映射关系

SO Leader (5 关节)              SO Follower (6 关节)
┌─────────────────┐             ┌─────────────────────┐
│ shoulder_pan    │ ──────────→ │ shoulder_pan        │
│ shoulder_lift   │ ──────────→ │ shoulder_lift       │
│ elbow_flex      │ ──────────→ │ elbow_flex          │
│ wrist_flex      │ ──────────→ │ wrist_flex          │
│ wrist_roll      │ ──────────→ │ wrist_roll          │
│                 │             │ gripper (独立控制)    │
└─────────────────┘             └─────────────────────┘

操作者拖拽 Leader 臂 → 关节角度被读取 → 直接映射到 Follower 对应关节

9. CLI 命令行接口

lerobot-teleoperate \
    --robot.type=so101_follower \
    --robot.port=/dev/tty.usbmodem58760431541 \

9.3 CLI 参数解析机制

使用 . 分隔的嵌套参数语法（由 draccus 提供）：

  • --robot.type=so101_follower → 设置 config.robot 的具体子类类型
  • --robot.port=/dev/xxx → 设置 robot 配置中的 port 字段
  • --robot.cameras="{...}" → 设置 robot 的相机配置（YAML/JSON 内联语法）

核心设计原则

  1. 关注点分离：配置（Config）、设备抽象（Robot/Teleoperator）、处理管道（Processor）、控制循环（Loop）各司其职
  2. 开闭原则：通过抽象基类和工厂模式，新增机器人/遥操作器类型无需修改核心循环代码
  3. 管道可组合：处理器步骤可以自由组合，适应不同机器人对之间的映射需求
  4. 精确时序：使用 precise_sleep 而非 time.sleep 保证控制频率的稳定性
  5. 优雅降级：try/finally 保证资源总是被释放，KeyboardInterrupt 被静默处理
